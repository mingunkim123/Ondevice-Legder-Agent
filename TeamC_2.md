# C팀 팀원 2 업무 실행 가이드

이 문서는 C팀 팀원 2가 담당하는 홈 화면 및 통계성 기능을 실제 개발 업무로 쪼개어 정리한 실행 가이드다.  
기준 문서는 `README.md`, `CLAUDE.md`, 그리고 C팀 업무분장 초안이다.

---

## 1. 내가 맡은 역할 한눈에 보기

팀원 2의 핵심 역할은 **홈 화면에서 필요한 공개 API와 인기 집계 기반 기능을 만드는 것**이다.

| Function ID | 기능 | API | 우선순위 | 의존성 |
| --- | --- | --- | --- | --- |
| H-01 | 메인 홍보 배너 조회 | `GET /api/home/banners` | 필수 | 낮음 |
| H-03 관련 | 부스 클릭 로그 저장 | `POST /api/booths/{boothId}/clicks` | 보통 | 낮음 |
| H-03 | 오늘의 인기 부스 조회 | `GET /api/home/popular-booths` | 보통 | A팀 부스 데이터와 점진 연동 |
| H-02 | 현재 진행 중인 공연 조회 | `GET /api/home/current-performance` | 필수 | B팀 공연 데이터 필요 |
| 선택 | 홈 통합 조회 | `GET /api/home` | 후속 | H-01~H-03 응답 조합 |

### 내가 직접 구현해야 하는 것

- 홈 공개 API의 Controller / Service / DTO
- 필요한 경우 Entity / Repository / Flyway migration
- 클릭 로그 저장 구조
- 인기 부스 집계 로직
- 현재 시간 기준 공연 조회 구조
- 공통 `ApiResponse` 형식에 맞춘 응답
- 예외가 필요하면 기존 `ErrorCode` / `BusinessException` 패턴 재사용
- API별 테스트와 Swagger 설명 보강

### 내가 직접 붙들고 있지 않아도 되는 것

- 부스 원천 데이터 생성과 운영 로직 자체
- 공연 원천 데이터 생성과 운영 로직 자체
- 예약률 산식의 최종 확정
- 타 팀 테이블이 아직 없을 때의 최종 연동 완성

다만 H-02와 H-03은 나중에 A팀/B팀 데이터와 자연스럽게 이어져야 하므로, **초기 설계부터 교체하기 쉬운 구조**로 잡는 것이 중요하다.

## 2. 추천 이슈 및 PR 분리 방식

README와 CLAUDE 규칙상 작업은 Linear 이슈를 먼저 만들고, 이슈 기준으로 브랜치와 PR을 관리한다.  
팀원 2 업무는 아래처럼 나누는 것이 가장 깔끔하다.

| 권장 순서 | 권장 Linear 이슈 | Function ID | 결과물 |
| --- | --- | --- | --- |
| 1 | `[H-01] 메인 배너 조회 API 구현` | H-01 | 배너 조회 API |
| 2 | `[H-03] 부스 클릭 로그 저장 API 구현` | H-03 관련 | 클릭 로그 저장 API + migration |
| 3 | `[H-03] 오늘의 인기 부스 조회 API 구현` | H-03 | 집계 API |
| 4 | `[H-02] 현재 진행 중인 공연 API 1차 구현` | H-02 | 응답 계약 + 임시/실데이터 조회 구조 |
| 5 | `[H-02/H-03] A팀/B팀 연동 보완` | H-02, H-03 | 부스/공연 실제 데이터 연동 |

### 원칙

- **기본은 1이슈 = 1브랜치 = 1PR**
- H-03은 `클릭 로그 저장`과 `인기 부스 조회`가 성격이 다르므로 가능하면 PR을 분리
- H-02는 B팀 데이터가 아직 준비되지 않았다면 `응답 구조 선반영 PR`과 `실데이터 연동 PR`을 분리
- 여러 기능을 한 번에 넣는 큰 PR은 리뷰가 느려지고 협업 충돌 가능성이 커지므로 피한다

## 3. 작업 시작 전 체크리스트

각 기능을 시작하기 전에 아래 순서를 따른다.

1. Linear 이슈를 먼저 만든다.
2. 이슈 키 `BACK-XX`와 Function ID를 기록한다.
3. Linear의 **Create branch**로 생성된 브랜치명을 확인한다.
4. 브랜치는 반드시 `dev`에서 시작한다.
5. 브랜치명은 영어여야 한다.
6. 구현 전에 API 응답 필드와 타 팀 의존성 여부를 짧게 확인한다.

예시:

```text
이슈 키: BACK-42
Function ID: H-03
브랜치: feature/BACK-42-popular-booths-api
```

## 4. 구현 로드맵

### 4-1. Step 1 - H-01 메인 배너 조회 API

#### 목표

홈 화면에서 노출할 배너 목록을 반환한다.

#### 작은 단계

1. 배너 응답 계약을 먼저 정한다.
   - 최소 후보: `id`, `imageUrl`, `linkUrl`, `displayOrder`
   - 운영 정책이 필요하면 `isActive`, `startAt`, `endAt`도 검토
2. 저장 방식을 확인한다.
   - 임시 고정 데이터인지
   - DB 조회인지
3. `domain/home` 또는 프로젝트 관례에 맞는 홈 전용 패키지 구조를 정한다.
4. `HomeController`와 `HomeService`를 만든다.
5. `BannerResponse` 같은 응답 DTO를 만든다.
6. DB 기반이면 `Banner` Entity / Repository / Flyway migration을 함께 만든다.
7. 응답을 `ApiResponse.success(...)`로 감싼다.
8. 목록 정렬 기준을 확정한다.
   - 보통 `displayOrder ASC`
9. 테스트를 작성한다.
   - 정상 응답
   - 노출 대상 필터링이 있다면 필터링 케이스

#### 완료 기준

- `GET /api/home/banners`가 정상 응답한다.
- 응답 배열 순서가 예측 가능하다.
- 공통 응답 포맷을 지킨다.
- Swagger 또는 API 명세에서 프론트가 바로 참고할 수 있다.

#### 추천 커밋 분리

1. `feat: H-01 배너 조회 API 기본 구조 추가 (BACK-XX)`
2. `feat: H-01 배너 조회 데이터 모델 및 정렬 로직 추가 (BACK-XX)`
3. `test: H-01 배너 조회 테스트 추가 (BACK-XX)`

#### PR을 올리는 시점

- H-01 응답 스펙이 정리되고
- 테스트 또는 수동 확인이 끝났고
- API 하나가 독립적으로 리뷰 가능한 상태가 되었을 때 바로 PR을 올린다.

### 4-2. Step 2 - 부스 클릭 로그 저장 API

#### 목표

부스 상세 진입 이벤트를 저장하여 H-03 인기 부스 집계의 원천 데이터를 만든다.

#### 작은 단계

1. 클릭 로그에 저장할 필드를 정한다.
   - 최소 후보: `id`, `boothId`, `clickedAt`
   - 필요 시 `visitorKey`, `sessionId`, `ipHash` 등은 팀 합의 후 추가
2. `BoothClickLog` Entity를 만든다.
3. `BoothClickLogRepository`를 만든다.
4. Flyway migration을 작성한다.
   - 테이블 생성
   - `booth_id`, `created_at` 또는 `clicked_at` 조회용 인덱스 검토
5. `POST /api/booths/{boothId}/clicks` API를 만든다.
6. 부스 존재 검증을 할지 정한다.
   - 현재 레포에는 부스 도메인이 있으므로, 존재 검증은 A팀 구조와 충돌이 없는지 확인 후 적용
7. 응답 형식을 정한다.
   - 데이터 없는 성공이면 `ApiResponse.successEmpty()`
8. 저장 실패나 잘못된 `boothId`에 대한 예외를 정리한다.
9. 테스트를 작성한다.
   - 저장 성공
   - 존재하지 않는 부스 처리 여부

#### 완료 기준

- 클릭 API 호출 시 로그가 저장된다.
- 집계에 쓸 최소 데이터가 DB에 남는다.
- migration이 함께 올라간다.
- 이후 H-03 인기 집계가 이 테이블을 기반으로 확장 가능하다.

#### 추천 커밋 분리

1. `feat: H-03 부스 클릭 로그 엔티티와 마이그레이션 추가 (BACK-XX)`
2. `feat: H-03 부스 클릭 로그 저장 API 구현 (BACK-XX)`
3. `test: H-03 클릭 로그 저장 테스트 추가 (BACK-XX)`

#### PR을 올리는 시점

- 클릭 로그 저장이 단독 기능으로 완결되면 바로 PR을 올린다.
- 인기 부스 조회 API까지 기다렸다가 한꺼번에 묶지 않는다.

### 4-3. Step 3 - H-03 오늘의 인기 부스 조회 API

#### 목표

오늘 기준으로 클릭 수가 높은 부스를 조회한다.  
초기에는 `boothId`, `clickCount` 중심으로 시작하고, 이후 부스명과 썸네일 등은 A팀과 연동한다.

#### 작은 단계

1. 집계 기준 시간을 확정한다.
   - "오늘"의 기준은 서버 시간대 정책에 맞춘다.
   - 이 레포 운영 시간대 기준을 팀과 맞춘다.
2. 조회 범위를 정한다.
   - 예: 오늘 00:00:00 ~ 현재
3. 상위 몇 개를 줄지 정한다.
   - 예: TOP 3 또는 TOP 5
4. Repository에서 집계 쿼리를 만든다.
   - `GROUP BY boothId`
   - `COUNT(*) DESC`
   - `LIMIT`
5. 응답 DTO를 만든다.
   - 1차: `boothId`, `clickCount`, `rank`
   - 연동 후: `boothName`, `organizationName`, `thumbnailUrl`, `mainMenu`
6. API를 만든다.
   - `GET /api/home/popular-booths`
7. 데이터가 없는 날의 응답 정책을 정한다.
   - 빈 배열 반환이 자연스럽다.
8. 동률 처리와 정렬 보조 기준을 정한다.
   - 예: `clickCount DESC`, `boothId ASC`
9. 테스트를 작성한다.
   - 순위 정렬
   - 집계 기간 적용
   - 결과 개수 제한

#### 완료 기준

- 오늘 기준 인기 부스가 정렬되어 반환된다.
- 데이터가 없으면 안정적으로 빈 배열을 준다.
- 추후 A팀 부스 상세 연동을 붙일 수 있는 응답 구조다.

#### 추천 커밋 분리

1. `feat: H-03 인기 부스 집계 쿼리 추가 (BACK-XX)`
2. `feat: H-03 인기 부스 조회 API 구현 (BACK-XX)`
3. `test: H-03 인기 부스 집계 테스트 추가 (BACK-XX)`

#### PR을 올리는 시점

- 클릭 로그 저장 PR이 먼저 머지되었거나, 같은 브랜치 의존성이 명확히 해소된 뒤
- 인기 부스 API가 독립적으로 동작하고
- 집계 결과를 설명할 수 있는 테스트가 갖춰졌을 때 PR을 올린다.

### 4-4. Step 4 - H-02 현재 진행 중인 공연 조회 API

#### 목표

현재 시간 기준으로 진행 중인 공연을 반환한다.

#### 작은 단계

1. B팀 공연 데이터 구조를 먼저 확인한다.
   - 공연 시작 시각
   - 종료 시각
   - 팀명
   - 장소
   - 썸네일 또는 대표 이미지
2. 아직 테이블이 준비되지 않았다면 응답 계약부터 먼저 확정한다.
   - 예: `performanceId`, `title`, `stageName`, `startsAt`, `endsAt`
3. 1차 구현 방향을 선택한다.
   - Mock/임시 데이터 반환
   - 인터페이스만 잡고 빈 응답 반환
   - 이미 B팀 테이블이 있다면 실조회
4. `GET /api/home/current-performance` API를 만든다.
5. 시간 비교 로직을 만든다.
   - `startsAt <= now < endsAt`
6. 진행 중 공연이 없을 때 응답 정책을 정한다.
   - `null`보다 빈 데이터 또는 명시적 상태 DTO가 더 다루기 쉬운지 프론트와 맞춘다.
7. 테스트를 작성한다.
   - 현재 공연 있음
   - 현재 공연 없음
   - 경계 시각

#### 완료 기준

- 진행 중인 공연 조회 API의 계약이 정리된다.
- B팀 실제 테이블이 들어와도 Service 내부 구현만 바꾸면 되도록 되어 있다.
- 시간 비교 기준이 테스트로 보호된다.

#### 추천 커밋 분리

1. `feat: H-02 현재 공연 조회 응답 계약 추가 (BACK-XX)`
2. `feat: H-02 현재 공연 조회 서비스 구현 (BACK-XX)`
3. `test: H-02 현재 공연 시간 조건 테스트 추가 (BACK-XX)`

#### PR을 올리는 시점

- 응답 스펙과 기본 시간 로직이 합의되면 1차 PR을 올린다.
- B팀 연동이 별도로 남아 있다면 그 작업은 후속 이슈와 후속 PR로 분리한다.

### 4-5. Step 5 - 홈 통합 조회 API가 필요해질 때

초안에는 `GET /api/home`도 포함되어 있다.  
이 API는 H-01, H-02, H-03을 한 번에 묶는 **조합형 API**에 가깝다.

#### 구현 기준

- 개별 API가 먼저 안정화된 뒤 진행한다.
- 프론트가 초기 진입 시 1회 호출을 선호하는지 확인한다.
- 내부적으로는 개별 Service 로직을 재사용한다.
- 이 API를 먼저 만들고 개별 API를 나중에 만드는 순서는 권장하지 않는다.

## 5. 언제 커밋할지

CLAUDE 규칙에 따라 수정사항은 **논리적 단위별로 커밋을 쪼개고, 커밋 직후 바로 푸시**한다.

### 커밋 타이밍 원칙

아래 중 하나가 끝날 때마다 커밋한다.

- 새 Entity와 migration이 함께 완성되었을 때
- 새 API의 Controller / Service / DTO가 한 덩어리로 동작할 때
- 집계 Repository 로직이 완성되었을 때
- 테스트 한 묶음이 끝났을 때
- 문서나 Swagger 보강이 별도 의미를 가질 때

### 커밋 전 반드시 확인할 것

```bash
git status
git diff --staged
```

확인 포인트:

- 이번 커밋 목적과 무관한 파일이 섞이지 않았는가
- migration과 Entity가 서로 맞는가
- 테스트 파일이 빠지지 않았는가
- 다른 팀 도메인 파일을 의도치 않게 수정하지 않았는가

### 커밋 메시지 예시

```text
feat: H-01 배너 조회 API 구현 (BACK-42)
feat: H-03 클릭 로그 저장 API 구현 (BACK-43)
test: H-03 인기 부스 집계 테스트 추가 (BACK-44)
docs: H-02 공연 연동 메모 보강 (BACK-45)
```

## 6. 언제 PR을 올릴지

### 기본 원칙

PR은 **기능 하나가 리뷰 가능한 상태가 되었을 때 바로 올린다.**

PR을 늦게 올리는 기준이 아니라, 아래 조건이 채워졌는지로 판단한다.

1. 해당 이슈의 완료 조건을 만족한다.
2. 테스트를 실행했거나 수동 검증을 마쳤다.
3. migration이 있다면 함께 포함했다.
4. README/CLAUDE 규칙상 필요한 협업 정보가 정리되었다.
5. PR 템플릿을 정확히 채울 수 있다.

### 팀원 2 권장 PR 타이밍

| 기능 | PR 시점 |
| --- | --- |
| H-01 | 배너 조회 API 단독 완료 직후 |
| 클릭 로그 | 저장 API와 migration 완료 직후 |
| H-03 | 집계 API가 클릭 로그 데이터 기준으로 동작하는 시점 |
| H-02 | 응답 계약 + 시간 로직이 안정된 1차 구현 완료 시점 |
| 타 팀 연동 | A팀/B팀 실제 데이터 결합이 끝난 시점 |

---

## 7. PR 작성 규칙 실전 요약

PR을 올릴 때는 `.github/PULL_REQUEST_TEMPLATE.md` 구조를 그대로 따른다.

### 반드시 지킬 것

- 관련 이슈는 `closes #이슈번호`
- PR 제목은 `feat: ...`, `fix: ...`, `docs: ...` 형태
- PR 제목에 이슈 키를 괄호로 중복 표기하지 않는다.
- PR 본문은 한국어
- Function ID 명시
- 실제 수행한 테스트만 체크
- 다른 팀 도메인 파일을 수정했다면 해당 팀 팀장 리뷰어 추가
- 각 팀 팀장 리뷰는 필수

### PR 전에 정리할 것

- `git log`로 커밋 흐름이 자연스러운지 확인
- 테스트 결과 정리
- 리뷰어가 봐야 할 쟁점 메모
- H-02/H-03처럼 타 팀 연동이 남은 경우, 이번 PR 범위와 후속 작업을 구분해 적기

## 8. 타 팀과 협업할 때 확인할 것

### H-03 인기 부스

A팀과 미리 맞출 것:

- 인기 부스 응답에 어떤 부스 필드를 담을지
- 클릭 수 외에 예약률을 함께 반영할지
- 예약률 반영 시 산식과 기준 기간
- 부스 상태값에 따라 홈 노출 제외가 필요한지

### H-02 현재 공연

B팀과 미리 맞출 것:

- 공연 시작/종료 시간 필드명과 시간대 기준
- 동시에 여러 공연이 진행될 수 있는지
- 현재 공연이 없을 때 프론트가 원하는 응답 형태
- 썸네일, 장소명, 출연팀 표기 규칙

## 9. 내가 따라가면 좋은 실전 순서

아래 순서대로 진행하면 충돌이 적고, 중간마다 PR을 올리기 좋다.

1. H-01 이슈 생성 → 브랜치 생성 → 배너 조회 API 구현 → 테스트 → PR
2. H-03 클릭 로그 이슈 생성 → Entity/migration → 저장 API → 테스트 → PR
3. H-03 인기 부스 이슈 생성 → 집계 API → 테스트 → PR
4. H-02 이슈 생성 → 응답 계약/시간 로직 → 1차 PR
5. A팀/B팀과 연동 조건이 맞춰지면 후속 이슈와 PR 생성

## 10. 최종 셀프 체크리스트

- [ ] Linear 이슈를 먼저 만들었다.
- [ ] 브랜치는 Linear 자동 생성명을 사용했다.
- [ ] `dev`에서 분기했다.
- [ ] Function ID와 이슈 키를 커밋/PR에 반영했다.
- [ ] 공통 `ApiResponse` 형식을 지켰다.
- [ ] 필요한 migration을 함께 작성했다.
- [ ] 기능별 테스트를 최소한의 수준으로 추가했다.
- [ ] 논리 단위별로 커밋했고, 커밋 직후 푸시했다.
- [ ] PR 템플릿을 빠짐없이 채웠다.
- [ ] 타 팀 도메인 파일을 수정했다면 해당 팀 리뷰어를 추가했다.
